# GB28181 P4a (full transport + ZLM control) — manual smoke test

Prereqs: a real GB28181 camera registered to nvr (P1-3 running with
`NVR_GB_ENABLE=1 …`), configured as an nvr device with `input_type=gb28181` and
`input_value={"device_id":"<gb>","channel_id":"<gb>"}`. Its stream mapping is
registered at startup (`stream_id` = the **nvr device id**), defaulting to UDP.
For TCP tests the camera must actually support that TCP role (passive: it dials
out to us; active: it listens and we dial it).

**⚠️ Transport behaviour is validated HERE on real hardware, not by unit tests.**
The automated tests use a fake receiver; they prove the bridge threads the
transport and does the two-phase active connect, but NOT that ZLM's `RtpServer`
actually receives PS/RTP over each mode. If a mode won't pull, the fix sites are
`ZlmControl`/`handler_zlm_cmd` (`nvr/src/zlm/cmd.rs`), `mode_for`, and the crate
SDP (`crates/gb28181/src/sdp.rs`).

Observe pulls via the log line:
`gb28181: pulling <dev> channel <chan> -> stream <id> (<Transport> port <p>)`.

1. **UDP pull (default):** `POST /api/gb/play {"device_id":"<nvr id>"}` (transport
   omitted → udp) → returns `{stream_id, url}`. Open `url`
   (`http://…:8553/rtp/<id>.live.flv`): the camera pushes PS/RTP to the UDP port
   ZLM opened; video plays. Log shows `(Udp port <p>)`.
2. **TCP-passive:** `POST /api/gb/play {"device_id":"<nvr id>","transport":"tcp_passive"}`
   → open `url`. ZLM listens on a TCP port; the camera connects in and pushes.
   Video plays; log shows `(TcpPassive port <p>)`.
3. **TCP-active:** `POST /api/gb/play {"device_id":"<nvr id>","transport":"tcp_active"}`
   → open `url`. After the INVITE 200, ZLM dials OUT to the camera's media addr
   (from the SDP answer, `y=`/`c=`/`m=`) within the 5 s connect budget. Video
   plays; log shows `(TcpActive port <p>)`. If the camera is unreachable, the pull
   errors ("tcp-active connect timed out"/failed) and the port is released — the
   UI stays usable and a re-open retries.
4. **Switch transport:** call `/api/gb/play` again with a different `transport`
   BEFORE re-opening the stream (let the previous pull tear down first — stop
   viewing so `on_media_no_reader` fires). The next open pulls in the new mode
   (confirm via the log's `(<Transport> …)`).
5. **Offline device:** stop the camera; opening the stream fails the INVITE
   (logged `pull for stream <id> failed`), nothing is left active, the port is
   released, and the UI stays usable. Re-opening after the camera returns pulls
   again.
6. **GB disabled:** with `NVR_GB_ENABLE` unset, `POST /api/gb/play` returns the
   "GB support is not enabled" error (no panic).
7. **Teardown:** stop viewing → last reader leaves → `on_media_no_reader` → BYE +
   the `RtpServer` port is released (`gb28181: released stream <id>`). No orphan
   port or dialog remains.

Notes:
- All ZLM control FFI runs on the dedicated `zlm-rtp` worker thread; a hung
  device can't block ZLM (the pull runs in a spawned task, and the active connect
  is bounded by a 5 s timeout).
- `POST /api/gb/play` only sets the transport on the mapping and returns the URL;
  the actual pull is lazy (fires on first view via ZLM's `on_media_not_found`).
- Media-status introspection (`GET /gb/streams`, live/receiving state) is **P4b**,
  not covered here.
